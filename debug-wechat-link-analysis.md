# Re-Read Debug Session

## Bug

- 症状：输入公众号链接后，任务可以运行，但前端看起来没有展示该链接对应文章的真实解读结果。
- 期望：任务完成后，结果页应展示该链接对应文章的标题、来源、时间，以及基于真实正文生成的解读内容。

## Hypotheses

1. 前端提交的链接与后端收到的 `articleUrl` 不一致，导致后端分析了错误输入。
2. 后端已经抓到正文并生成了新报告，但前端轮询或取报告详情时拿到的是旧报告/样例报告。
3. 后端真实模型返回成功，但报告渲染层仍主要复用静态样例结构，视觉上像“没解读这篇文章”。
4. 前端页面状态在任务完成后没有完整覆盖本地 state，导致标题或部分字段更新了，但主要 sections 仍来自旧 state。
5. 链接抓取成功但模型输出字段过于通用，结果虽是真实生成，但没有足够强的文章指纹，造成“看起来不像这篇文章”的错觉。

## Plan

1. 对前端提交、后端接收、任务完成、报告渲染四个节点做最小化日志埋点。
2. 用用户给出的公众号链接复现一次，采集前后端证据。
3. 根据证据排除或确认上述假设。
4. 在证据明确后再做最小修复。

## Evidence

- 已通过后端直接验证用户链接，抓取结果回填为：
  - 标题：`Nature : AI 自己跑完了一轮科学：从查文献到验证老药，30 分钟发现了一个治失明的新策略`
  - 来源：`AI Crossfields Insights`
  - 日期：`2026-05-29`
- 已通过后端验证真实模型链路，报告首段 `highlight` 为 `kimi`，说明不是 mock。
- 已读取生成后的报告内容，发现：
  - 标题和来源已经是当前文章
  - 但 `sections` 仍保留了样例模板里的“缺失说明与等待建议”等静态内容

## Conclusion

- 假设 1：排除。链接已正确传入并抓取成功。
- 假设 2：基本排除。后端生成了新的 `reportId`，不是旧报告复用。
- 假设 3：确认。真实模型结果存在，但主体 sections 仍复用静态样例模板。
- 假设 4：暂不成立。主要问题出在后端报告装配，而不是前端 state 没切换。
- 假设 5：部分成立。模型结果本身较短，但真正误导用户的是样例 sections 没被替换。

## Additional Evidence

- 通过预览容器控制台观察到大量 `reload, previewUrl = ...&v=N` 日志，说明预览页发生了整页重载，而不是应用内跳转。
- 后端每次生成报告都会改写项目内 `backend/data/generated-reports.json`。
- 该文件位于项目目录内，极可能被预览服务监听并触发热重载。

## Additional Conclusion

- “一瞬间刷新回首页” 的直接原因更可能是预览环境监听到项目文件变更后主动重载。
- 最小修复策略：将运行时持久化文件移出项目目录，避免每次生成报告都触发预览重载。
